FFT Randomizer

FFT Randomizer

This JSFX script is an unique spectral randomizing tool to shape up any sound's both magnitudes and phase independently and dynamically, in a random way that just looks and sounds cool! It comes with similar settings as "cloud fractal noise" algorithms in imaging software (such as the layered cloud generation) but instead of generating images, you sculpt up the spectrum's both magnitude and phase with it. In effect, it's inspired by the Diamond-square algorithm but it is not exactly the same, as it also allows you total control over each of the layers.

Please note that this is not a conventional FFT randomizer/processor. Most of those FFT-based tools deal with FFT bins directly, for example by randomizing them all on each window, which is just crappy and doesn't have many uses in my opinion, not to mention the drastic difference in effect when changing the FFT window size. That's not what this tool is about, at all! It uses mathematical models that are completely independent of the bins. The only time it affects the bins is at the end of the processing chain, when they are simply interpolated and smoothed based on the model calculated prior. Think of it as having a mathematical model that gets applied to pixels on an image with values between pixels interpolated — instead of randomizing each pixel itself, which is what standard and simple FFT tools would do. The latter is obviously bad, because its effect depends on the resolution of the image itself and is very limited in effect (it only makes high frequency noise!).

As a result, the output on the sonogram can look like "clouds" if used on white noise input, but it works on any signal to shape it; it's intended to be used on any generic material and alter its magnitude and phase by the "cloud mask".

It has two versions, Stereo and Mono, which are pretty obvious in function. The Mono version has almost half the CPU cost, so use it if you can on mono material (in fact, you'll probably use this to sculpt sounds so most likely you will use the Mono version more often). Please note that the Mono version requires the Stereo version to be available and placed under the FFT Tools directory (in your Effects folder). If you put them somewhere else, please change this script's import accordingly (any text editor will do, just change the path).

Features:

  • Filter any input sound randomly and dynamically by changing its magnitudes and phases, independently.
  • Layer-based random shape algorithm that changes with time, so you can have smooth shapes, or more noisy shapes, and have them slow or fast independently.
  • Exact and independent control over each layer, even including a spectral graph to change the layer's effect on the output, depending on frequency.
  • Special "Additive Mode" on phase amounts to have all the phases changed relative to a new "center", so the entire phase spectrum will shift, but randomly.
  • Ability to use special Program Change events as triggers to "reset" the shape and get a new one, within a time specified.

—LATENCY— This script, like most effects dealing with FFT, adds a latency in samples equal to the FFT Size. I did put a delay compensation for it, so it is reported, but it doesn't work in ReaJS (it should work in REAPER's native JS, though). So if you use it in another host via ReaJS, you'll have to manually compensate for this latency by delaying the other signals yourself, or at least keep this latency in mind! To convert from latency in samples to latency in seconds, simply divide the FFT Size by your sample rate. So for example if you work in 48000Hz, a FFT Size of 8192 means a latency of 0.1706… in seconds (170.6 ms). Increasing the FFT Size tends to decrease CPU use slightly because there's less calculations to be done per second (in the spectrum changes) while also increasing frequency resolution, but also increases latency and decreases time resolution (fast changes in the spectrum are more blurred).

—SAMPLERATE— In case the sample rate is 88200Hz or above, then only the first half of the spectrum is processed. That's because this effect was made to work on a full spectrum only with sample rates below 88200Hz, by design. For instance, a sample rate of 48000Hz processes the full spectrum, while a 96000Hz sample rate processes only up to the 24k bin (so basically, same as the 48k sample rate). Similar for 88200Hz, where it will output only up to 22050 frequencies because it's half of its entire range (in other words, it will be the same as 44100Hz). Once again, this is on purpose to keep the effect concentrated within the standard range. You'll have to modify it slightly if you don't want this behavior (just search for 88200), but keep in mind that the larger the spectrum that is processed, the less detail you have available in the layer frequencies, since the layers will be stretched for the entire spectrum and you have the same amount of layers…

—WARNING— There will be glitches when changing the FFT Size or sample rate while it's processing. Also it wasn't really made to be automated, but you can do it. However the performance may be slightly worse and you will hear glitches if you automate the FFT Size. So just don't touch it with automation, you can automate other parameters. I didn't really bother to optimize the @slider section much, but the @sample and @block are heavily optimized, so the code can be quite a mess there (in fact, if the code seems hardcoded, it's due to that, sorry; for me one of the priorities was to have it used and fast enough, understanding its code came second).

—TIP— I heavily recommend you use a real-time Sonogram (or Spectrogram) VST when learning or even using the FFT Randomizer! That's very important so you can see its effects and beauty. Personally, I use Sonogram SG-1 by ag-works. Yes it is old and not maintaned but it works great and is absolutely excellent! Plus it's free. I cannot recommend it enough, really.

Quickstart

Ok I get it you're confused what this actually does, and since I'm too lazy to make a video, here's how to see it for yourself without understanding a thing yet, in a few steps!

  • Step 1: Use a Spectrogram/Sonogram like the one mentioned (Sonogram SG-1), insert it after this effect, and keep it open (to look at it).
  • Step 2: Send white noise to this effect, it should show up as white noise on the Sonogram.
  • Step 3: Reduce the Gain (dB) so it won't blow your speakers, say set it to -20.
  • Step 4: You should have Layer 1 selected in the drop-down by default. Then adjust the Magnitude Range (dB) to an amount that is less than the Gain (dB), say to 16.
  • Step 5: Increase the Magnitude Speed (Hz) so the changes start to happen faster, make it about 1 Hz or so to be smooth, since we're on Layer 1.
  • Step 6: You should see patterns in the sonogram on the noise! Your white noise is no longer boring and static, and it varies over time: its magnitude over different frequencies is being changed. Try play with Magnitude Contrast and see what you get! What is the difference? Increasing the contrast clearly makes the variations more sharp, right? What about decreasing it?

And remember this is just one layer, try changing the layer from the drop-down menu and see the difference it makes (don't forget to also set that new layer's respective parameters, as all of the above are specific to each layer, except for Gain (dB)). Also remember that, while white noise is a good way to see and learn how this works, and even during designing the parameters and setting it up, the effect can actually be used on any input! Please don't forget to turn Gain (dB) lower if you add multiple layers with more dB range, though.

Parameters

There are many parameters in this script. So many, in fact, that you cannot even control all of them at once. Most of the parameters you see are for one layer — the rest are not accessible until you change the Edit Layer view, which is a drop-down parameter that specifies which layer is the one being edited. The first 3 parameters, though, are for the entire script and not layer-specific:

  • FFT Size: The size of the FFT Window used for processing. This script uses 4× Overlap with a Blackman window. Note that small FFT sizes have much more time-resolution and can work with much faster speeds/randomizations in time, while losing some frequency resolution (the frequency-specific modifications aren't so smooth). Unfortunately it seems like small FFT sizes also use up more CPU than high ones, even though the actual FFT is faster for smaller windows. This is because on each FFT, the randomized spectra values are calculated and smoothed: this is what enables the increased time resolution for small FFT sizes. By time resolution I obviously mean the fact that this script changes the spectrum (both magnitude and phase) over time, so the resolution here refers to how fast it can change between two such shapes. Otherwise, the FFT nulls out with original, so there's no quality loss with any FFT Size. It's only because the FFT randomized spectra change with time that it matters, and that is why the 4× Overlap is needed, due to the changes in time of the things that alter the spectra. Note that the FFT Size is directly related to the latency!
  • Gain (dB): A simple post-gain (which is very useful for the EQ, explained way below), because the sound can get out of bounds easily with this script, and also because it comes completely free in computation, as it gets pre-computed in another variable that gets multiplied anyway. So no harm in adding it.
  • MIDI Force Change (ms): This defines how fast the spectrum "resets" when it receives the special MIDI Program Change trigger, which is explained below. It basically "speeds up" the entire spectrum so that it changes the entire spectrum in the time specified here. For example, if this setting is set to 100ms, then once it receives the trigger, it will speed up so that its spectra changes completely within 100ms, effectively forcing a change in the spectrum (even if it is normally way slower over time with changes). This can be very useful if you trigger a new sound and don't want the old shape to carry over, makes it less predictable and more fun. An example is provided about this later in its section, when you understand more about this script's inner workings…
  • Edit Layer: Selects which layer to edit and be in view. This parameter doesn't do anything by itself, all it does is enable you to change other layers. You will notice when switching layers that the rest of the parameters also get changed to the new layer's values. All changes done will be stored for those layers, so just use it as a sort of "tab switcher"… Please note that the graph at the bottom will also change if it's a layer-specific graph (the equalizers are not, both magnitude and phase).

The rest of the parameters are layer-specific. Before I dwell into them, you're probably thinking "What the hell is this layer talk all about?" — so I will explain it now. Basically, you have 7 layers, and you can set them up independently. Each layer has a specific number of "features" or "hills" (or "random points" to be technical about it), and each of those is randomized, and then smoothed. Obviously, since each layer occupies the same spectrum range, this means that layers with less "features" or "points" will be more smooth, because between those points there is a smooth interpolator. Imagine looking at a blurry or pixellated image — those are the layers with the fewest points.

Layer 1 has the biggest (but fewest) features: 8 in total, spanning the spectrum (and 1 extra one at the beginning, for pre-smoothing). Layer 2 has double the amount of normal points, so it stands at 16 (but still with 1 extra). Layer 3 has 32+1, and so on. The last layer, which is Layer 7, has 512 features (and 1 extra). So at most, you can split the entire spectrum into 512 randomized features (and an extra one for pre-smoothing) with Layer 7. Each Layer doubles the amount of features of the previous while halving their size, because obviously twice as many features fitting into the same spectrum range means each feature's size is smaller. Every layer is of course smoothed first, then added to the others to produce the result.

Each Layer has two types: Magnitude and Phase. For each type, you set how much it should be able to affect and alter at most (randomly between -amount and +amount, which depends on the type); for magnitude it's in dB, while for phase it's in degrees. It's recommended you use low magnitude amounts for subtle effects, and that higher Layers have smaller magnitudes set than the lower ones. This is not a rule that you need to follow, more like a guideline to make it look more natural, it's up to you though. Then you set the speed in Hz at which new spectra for the features should be produced for each layer (so each layer can evolve at a completely different speed!). A layer is simply a "spectrum shape" of values that are random and spaced equally apart (e.g 8+1 for Layer 1), smoothed with linear interpolation then Moving Averaged to make it a quadratic process.

So with the basics of layers out of the way, let's look at the parameters for each layer:

  • Magnitude Range (dB): The layer's maximum magnitude amount it can modify, in both directions. So a range of 6dB means it can have points ranging from -6dB to +6dB in effect, which will then change the magnitude of the signal being processed. Obviously, the actual amount will be random between the two limits, but this parameter is basically telling the layer how strong it is overall. It is basically the "amplitude" of the points being generated, for magnitude.
  • Magnitude Contrast: Changes the random distribution of the points within the above range. The default value of 0 makes it completely uniform, so every point between the range has equal probability of being chosen. Decreasing the contrast results in values near the middle (which is no magnitude change) being much more probable, with the extremities (±range dB) being less probable — the lower the contrast, the stronger this change is. Likewise, increasing contrast leads to the opposite effect: values near the middle will be much less likely than values at the extremities, the further away from zero the contrast is.

    Now, setting the contrast to negative values will make the layer have less and less impact, since most of the values will be around the middle (which is no magnitude change), but occasionally you will get a point that's far away and thus it will be different than just lowering the Magnitude Range — it will be rare though, which is surprisingly what happens in some natural noise patterns. On the other hand, positive contrast will make most features very visible and less natural, so the randomness will look much more sharp, but if you want it for effect or a specific combination, feel free to use it!
  • Magnitude Speed (Hz): Simply defines how fast the layer evolves over time, for magnitudes. Note that the speed cannot be zero (if it is, it will bug out), so it always evolves even if it's excruciatingly slow — if you want a more static spectrum then set the speed low enough that the changes aren't noticeable. Note that fast speeds may add warbling effects to the sound, but it's all part of sound design right? Remember that even with low speeds or almost-static layers, you can still force them to change with the special Program Change trigger. If you need fast speeds, consider lowering the FFT Size which increases the time resolution (at the expense of frequency resolution).
  • Magnitude Speed Modulation Rate (%): Defines how much percent of the speed is the modulation's speed. To explain this, I'll have to introduce you the speed modulator internally. Basically, this script modulates the Magnitude Speed by a given factor — this is approximately up to 40% in both directions (slower or faster), but it is a random modulation over time. You don't change the amount of modulation with this parameter. Instead, you change how fast the modulator changes its value, based on a percentage of the speed above. For example, a 200% Rate simply means that the modulator changes at twice the speed of the actual Magnitude Speed. e.g if the speed was 5Hz, then the modulation change happens 10 times per second. So the speed itself gets altered 10 times per second. The modulation itself is linearly interpolated between successive values, giving it a random acceleration effect. This makes it less "static". If you don't understand it, just set it to arbitrary values, different ones for each layer, to give it a less predictable evolution over time.
  • Phase Range (Degrees): The layer's maximum phase amount it can modify, in both directions. So a range of 90 degrees means it can have points ranging from -90° to +90° in effect, which will then change the phase of the signal being processed. Just like the Magnitude Range, of course, the actual amount will be random between the two limits, but this parameter is basically telling the layer how strong it is with phase changes overall.

    But that's not all! Notice how you can have negative range here? You'd think that's strange, since the positive values mean that it can go in either direction already, right? Well, you're right, but the negative values here change the mode of the point generation process: it switches to the so-called Additive Mode. This works only for phases because phases wrap around, so it's not an accumulating nightmare that grows out of bounds. When you set it to negative values, it randomly selects one of the layer's points to start with, generating a random value there (its phase is controlled by the value of this range, without the negative sign). Then, instead of doing the same for every other point like a positive range would, it adds a random value to obtain the neighboring values. e.g: instead of an absolute random value between -90° and +90°, it adds to the previous value to generate a new one: hence "Additive Mode". The result is that the phase spectrum is centered around the new random point, instead of zero. It also randomly prefers one direction when doing so (i.e upwards or downwards) to make it look less uniform, but this direction is not 100%, it's a random chance.

    You should use the Additive Mode on very few layers, perhaps just one. This is if you want the phase changes to not be centered around the normal phase (0°; unchanged) or any other single point, which is what normal modes would do without an additive layer. It keeps the phase randomized dynamically, instead of static only from the Phase Equalizer. But note that you should use it on only one Layer. There's no point using it on more than at most 2. This is because once the "dynamic center" point is made with this additive mode, all other layers should be normal modes (positive ranges) and will alter the phase around this (since they just add to it). For this reason, it's recommended to use the Additive Mode on a layer with a somewhat big phase amount, and slightly lower the contrast, perhaps on Layer 1. Then, use the other layers to shape it more around this random-generated center.
  • Phase Contrast: Controls the random distribution of the points, the same way as the Magnitude Contrast really, except it does so for Phases.
  • Phase Speed (Hz): Simply defines how fast the layer evolves over time, for phases. It has the same principle as Magnitude Speed so I won't bother explaining it again. However, the effects of a fast speed here are different. Fast phase speeds can result in clicks or other crappy discontinuities, so you probably shouldn't set the speed here too high if your phases also happen to have big ranges, unless you're really after those artifacts.
  • Phase Speed Modulation Rate (%): Once again, same as its magnitude counterpart, except it controls the Phase Speed and is set in terms of that.

—NOTE— With very high ranges (especially in magnitude) and smaller FFT Size, there may be glitches at the very low end of the spectrum. Use the graph explained in the next section to lower the range amounts at the bottom end (or the EQ to get rid of those frequencies completely, I'm talking about frequencies below 100Hz here or so).

To recap: Each layer has a specific amount of random points spaced equally apart in the frequency range. Higher layers have more points, thus each point is smaller as they fit within the same frequency range. These random points have a defined random amount range, plus or minus, and they cannot get values outside this range. You can squeeze this range for each point towards the extremes or towards the center with the contrast, or leave it uniform (zero contrast). Each layer can evolve over time at a specific speed you set. It is interpolated with both linear interpolation and then moving averaged (smooth) in both the time direction (how it changes from one spectrum to another), and the frequency direction. This speed is also modulated over time randomly from -40% up to 40%, giving it an acceleration effect and making it less static. This random change of speed is linearly interpolated and happens at a percentage of the original speed, which is set by you, between 50% (half speed) to 200% (twice speed). Lastly, each layer has two types, magnitude and phase, that it can affect and which are independent. The phase type can also function in an Additive Mode where the random points are centered around a new random point and accumulated from there, instead of uncorrelated values on each point.

Got it? If you haven't, please re-read the entire section again, it is quite important to understand this part before moving forward.

As a personal note, randomizing phases is useful to add variety to sounds by making them less static, even if they sound similar. It also diffuses sounds in certain cases (most noticeable on transients), which again helps with giving life to boring samples or other static sounds (from synths with no dynamic randomization features). Most people will probably not find a use for it, but it was essential for me, being a perfectionist, and I was tired of nobody in the industry caring about creatively altering phases (unless it's to align them or other engineering task).

Graph

At the bottom of the UI, there is a graph — but the graph currently under view and edit is selected differently. Regardless, you can only see and edit one graph at a time. There are multiple types of graphs and can be set per-layer and per-type, but also global. The global graphs are the equalizers described in the next section. Here I will talk about the per-layer and per-type graphs.

The per-layer graphs are a way to individually control your "range amount" for each frequency point of each layer and type. Selecting a layer in the dropdown list will change the graph here for edit as well, unless you're into the equalizer graphs. Right-clicking on the graph toggles between magnitude and phase for the current layer being edited.

You can individually control each of the points on the layer's spectrum as a percentage of the actual amount, by changing how much the amount alters that frequency point. Values at the top of the graph (100%) mean that the entire amount applies to this point: in other words, it ranges from -amount to +amount, randomly. Less values, such as 50% (at the middle of the graph, vertically) halve the range. It's a percentage of the max amount used in the slider. So if you are in a magnitude graph and the Magnitude Range is 6dB, but you put the point at a specific frequency to 50% (middle), then that point will only get magnitude values between -3dB…3dB. Values of zero on the graph (at the bottom) mean that this point is always zero and not affected in any way by any randomness (the layer doesn't alter the sound at this frequency).

Obviously, a flat horizontal line in the graph doesn't serve much purpose, you could just as well decrease the range itself. The real power of the graphs comes from the fact that you can change the range itself based on the frequency. So if you want a layer to be strong only at the upper frequencies and not affect the lower ones, then you can just use the graph for it, even if you set the range high, if the graph is low at the lower frequencies, they won't be affected much (while the higher ones will).

To focus a graph without modifying it, simply hold ALT while clicking on the graph, or right-click on it twice (change type back and forth). Lower layers have way fewer points to edit on the graph. Layer 1 has only 9, Layer 2 has 17, Layer 3 has 33, and so on. The last layer, Layer 7, has 513 points. So it is recommended that you resize this script's UI to be at least 513 pixels wide, so you can control every point, otherwise they get linearly interpolated.

Modifying the graphs is a simple matter of just clicking around and drawing. I know this does not produce exactly smooth results, which is why there is a Smooth Edit available, in which you smooth the graph already made. By holding Shift and clicking, you smooth out the points with a crude quadratic-curve-fitting algorithm within the range shown (it shows in a background lighter than black). You can change this smooth range by holding Shift and using the Mouse Wheel. Just as well, you can change the smoothing amount (i.e how much it smooths in time) by using the Mouse Wheel but without holding Shift. You can see the smooth amount on the bottom-right of the display, which shows 3 things: the values at the specific mouse pointer (the vertical value, then frequency corresponding to it), and then on the right shows the smooth amount, 0…100. By default, this is 50. Note that the value 0 still smooths a bit, it's not "no smooth at all" (which would be quite pointless).

There is another way to change the smooth size and amount, if you want more precision and/or don't want to use the Mouse Wheel. By pressing Space you enter into this mode that changes both the size and amount of smoothing, by simply moving the mouse around. If you move the mouse horizontally, the further you move from the point where you pressed Space, the larger the smooth range will be (it is shown as you do it, don't worry). By moving the mouse vertically, from bottom to top, you increase the smooth amount: it is shown both as a number near the mouse pointer, and with the visual meter on the right, which should help you understand just "how much" smooth it is from 0 to 100 visually, with feedback. Once you are satisfied, press Space again and the new values will be used. If you change your mind or don't want to change them, simply press ESC and you will be back to normal drawing mode with the smooth range and amount unchanged.

Note that each graph here has 1 extra point as mentioned in the layer overview, this is for setting up the interpolation. It is the first point at 0Hz, which is used for pre-smoothing since the quadratic interpolator requires previous data. The last point in the graph is also "after" the actual frequency range, but it does affect the upper range since it skews the last part, so it is important. Due to this, the frequency display will vary depending on the number of points in the graph — in other words, it will be different for each layer. The last point will always be one half-point distance after the frequency range itself, but since the distance between points is different for each layer, the resulting frequency range is different as well. This should not be a problem at all, although sometimes you won't get the exact results you expect, it's simply due to how the smoothing algorithm works. I tried to make the frequency display on the bottom right as intuitive as possible, but remember it's not exact for all cases. Always use a sonogram to confirm you get what you want!

I hope it's not too complicated, it's really easy once you get the hang of it. I tried to make everything fit together in a logical way. Anyway, if you are wondering, the smooth function first linearly interpolates the points (since it's unlikely your smooth will fall exactly on points) using weights to fit a quadratic function: left and right points are unchanged, the middle is averaged between 3 points, with different (sinusoidal) weights, then the spectrum is just interpolated between this smooth quadratic curve and the actual values, depending how much time has passed. I guess that doesn't sound very easy, nevermind…

Equalizer Graphs

These graphs are different from the per-layer graphs above because they are global to the effect and not dependent on any layer. In fact, they are simply a post-processing of the entire 513-point data, so they are not technically real equalizers but serve almost the same effect anyway. The script already does the FFT and smoothly interpolates between the bins, so adding these equalizers comes at almost no cost at all — definitely much faster and easier than chaining another equalizer afterwards. Just like Layer 7's graph, these equalizers have 513-points, because the first one is for setting up the interpolation.

While the equalizers are not based on a layer, they do indeed have either type: magnitude and phase. You're probably not used to having a phase equalizer to control the phase surgically because I'm not aware of any EQ, FFT or not, that has one! I added it because I was annoyed how many developers ignore such a useful part (especially in randomization and sound sculpting). You can obviously control the phase relationship completely independent of the magnitude, it simply gets added at the end of the phase-mask that was randomly generated.

To switch between the equalizers and the layer graphs, simply press the Tab key while the graph is focused (if you need to focus the graph without changing anything, just hold the ALT key as previously mentioned, and then click on it). All the other things here work the same as the layer-based graphs. Right-clicking, as usual, switches between Phase and Magnitude equalizers.

Note that the Phase Equalizer's zero value is at the middle, because you can adjust the phase by up to ±180° (i.e in either direction). In contrast, the Magnitude Equalizer can only attenuate down to -42dB; it cannot boost at all. This is by design, to give you more precision when editing the Magnitude Equalizer, because you can boost the signal with the Gain (dB) parameter anyway. So if you want to boost the signal in certain parts only, simply modify the Gain (dB) parameter to boost all of it, then attenuate the parts you didn't want boosted (i.e do the inverse shape of what you wanted) in the Magnitude Equalizer.

MIDI Force Change

So, I was mentioning previously how you can force a "change" in the spectrum's shape with special MIDI Program Change events. Now that you hopefully understand how the whole layer-thing generation works, and how it evolves over time, you'll probably be able to figure this out easier…

I feel like an example would be appropriate now. Suppose you have this scenario: you want a randomized spectrum, you have all layers set up properly, but you don't want them to change much in time, if at all. You set the layer speeds very low (0.1 Hz or lower for instance), so the spectrum looks almost static and barely evolves. That's fine, but what if you want to retrigger a sound, with a new random spectrum shape? You don't want it to "carry over" the randomized shape of the last sound, but want a new one only when you trigger it. The spectrum still shouldn't change during the sound's lifetime, but once a new one is triggered (like a drum sound, for example), you'd want another spectrum so it sounds different than the previous one and looks different on a sonogram, too.

That's where this MIDI thing comes in. By sending Program Changes (on Channel 1 only, currently) to this script, you will force a change to the spectrum when that Program Change happens. It won't actually change any parameters, but will speed up the spectrum's evolution internally so much that the shape will be changed by the time the interval specified elapses. In other words, the time it takes from the moment the event happens until the new spectrum is fully available, is the one specified in the parameter MIDI Force Change. That is why it is in milliseconds, because it specifies how much it should take for the new spectrum to show up. So, if you retrigger some sound, just send a Program Change on Channel 1 before triggering the sound by approximately the same time interval as in the MIDI Force Change parameter. Now you'll get a new spectrum for the new sound, while also not having a too-abrupt discontinuity between whatever is left of the old sound (i.e tail) and the new one. That's the purpose of the MIDI Force Change setting, to make this transition somewhat smooth.

Note that the larger the FFT Size is, the less exact the transition's time will be relative to the one specified in the MIDI Force Change parameter. You may not be able to have the lowest setting properly, which is 25ms. In such case try upping it to 100ms or using a lower FFT Size which implies more time resolution (at the expense of frequency resolution). You should also use a higher MIDI Force Change interval if the difference between spectra can be quite big, else you'd get audible clicks or whatever from too fast changes…

By the way, this doesn't actually fade to a new spectrum. Instead, it speeds up the current spectrum so much, that over the course of the MIDI Force Change interval (give or take a few random percentage), it reaches a new spectrum. In other words if the time between two spectra is in the 0…1 range, then this simply does +1 to such time in that amount. Note also that if you have layers that have a fast speed, they might actually slow down when an event happens instead of speeding up, which makes them not look so good. So if you have layers with fast time and they are noticeable enough (big amounts), use a very small MIDI Force Change setting to make the transition fast. Small amounts work great for drums and other percussive or transient-based sounds with high attacks: just don't forget to delay the triggering of the sound by 25ms if you use a 25ms setting, and send the Program Change 25ms before triggering the sound.

As for the value in the Program Change event, you should set it to max (127). I might use it later for other stuff and don't want to break backward compatibility with it, so follow this guideline and use the value 127 for it, better be safe than sorry right?

Closing Words & Another Example

Well, after a request for a sawtooth demo, I realized I was lacking information on its use cases and other things I believed in when I made it. I actually ended up using it a lot in sound design right now and plan to for future shaping of textures, I'm sure it will be one of my most used effects overall (but maybe I'm weird, but that's besides the obvious needed ones that everyone uses, like EQ/compression/reverb of course). But first, here's another strong-effect example used on an Octaved Supersaw. Being a rich input source (supersaw doubled 1 octave below) enables it to be much more audible, since it has more input frequencies available to shape up. You can download the mp3 here from KVR. It uses the same settings as the previous Simple Example Showcase, but this time used on the Octaved Supersaw (with a bit of phase randomization added, probably inaudible).

I also used a slight compressor at the end of the chain, because this effect can get quite wild in dynamic range when used strongly like this! In fact, it would benefit from more compression for such "drastic" effects like this (42dB range for a single layer, the Magnitude Range (dB) setting), but I wanted to leave it more "raw" so people know what to expect, so didn't apply much compression to it.

Please note, however, that since it uses the FFT and not a Wavelet-based transform (not available in JSFX anyway), the shaping is linear, thus the majority of the effect will be in higher frequencies that most people consider "less important". This can be both a blessing and a curse. A curse because, well, most people care of the lower frequencies so they won't instantly find a use for this effect, but that's fine since I intended it to be more unique/experimental/not like anything out there, not to be super popular. They want more "instant wow" effect. However, it can also be a blessing because there are incredibly few plugins out there that allow you to create dynamic shapes in the higher spectrum with enough precision like this. That's one of the main reasons I made it, in fact. There's simply not much choice out there in the wild especially not free that does this.

For instance, it can work great on "simple" input like a simple synthesized instrument too, especially the phase randomization there helps randomly shape it up, but in those cases it will be very subtle in effect but that's okay! It adds "dynamic variation" to otherwise static sounds. If you look at the spectrogram/sonogram of a real acoustic instrument you will see such "shapes" and variations in its noise/breathiness/etc and in its harmonic/inharmonic tones. It is my firm belief that this is something overlooked by a lot of developers out there! They focus too much on what every other synth/effect does, on the strong audible things, to "wow" the audience. But looking at a synthesized sound versus a real sampled one, the lack of dynamic variations especially in the higher frequencies is very recognizable on the synthesized sound. Even if you can't hear it instantly/consciouscly much, you can instantly recognize a synthesized sound by looking at its spectrogram. That's sad.

The FFT Randomizer can help with that, too. A lot of things we hear are subconscious and not always immediately known "why" they give a certain vibe. I think part of the "magic" of real acoustic instruments is this subtle variation not just in the lower frequencies/phases but also the upper/higher ones. It makes the instruments more "alive". For example think of a bowed or wind instrument and its "breath" noise effect. It is very complex in the spectrum, not a static white/filtered noise at all. But most synthesized sounds use just filtered white noise. It's "static" and too simple (in shape). On the other hand, a real breath sound changes not just in amplitude/intensity, but in fact, its shape is constantly changing in time, and different on every frequency. Subtly but subconsciously, we "pick" this up, and makes us think the sound is more alive. The human voice is also incredibly complex in this "dynamic filtering". Synthesized voices always lack this integral effect, you can inspect their spectrograms to see it yourself, that's why they always "sound the same" when you make them sing a long note or repeat a melody/passage. They have no "life" to them. That's why synthesized choirs work better than synthesized voices to "fool" the audience: this dynamic filtering effect becomes less pronounced, because of the average random filtering of everyone in the choir, so it becomes more like "white noise" instead. The more random things you sum up, the more it becomes white noise, just fact of nature. So the more voices the synthesized choir emulates, the more it can sound like a real choir.

Of course with the FFT Randomizer you don't need to apply it just to the noise, you apply it to the whole thing! In those cases, "subtle" settings make the most sense. A range of 42dB in the shape is far from subtle (because it's ± so it's huge), for those cases you'd use 6-10dB per layer or less! And of course, phase randomizations too. Phases are mostly inaudible, but changes in phase add more dynamic variations (which is the point of the effect). Phase changes are actually very noticeable in transients (it diffuses them). For example if you use it on samples, you'd get some sort of free subtle round-robins because no samples will be the same! They'll differ slightly in spectral shapes and phases. Keep in mind for transients to mix this with the original if you do phase manipulations, otherwise you'll get a lot of diffused transients, which can be cool in its own way, but if you want to keep their punch I guess you'd rather not want that. :)

I'm not saying that the FFT Randomizer does the exact natural dynamic filtering found in acoustic instruments. It doesn't use physical modelling at all (even though most physically modelled instruments, apart from stringed ones, lack this dynamic variation too…), but it's inspired from purely artificial "imaging software algorithms". If you used GIMP/Photoshop, you know of these "cloud" effects that are used to make artificial textures, it's the same principle. Natural things tend to be… ahem, "random". :) Still, it helps to give some life into them, better than nothing!

Plus, you can use it experimentally and for sound FX design too. Like the two examples I gave both with the same settings. These are exaggerated effects, but that's what demos should be for I suppose, so people get hooked to its possibilities. In these cases you use exaggerated settings to make the effect strongly known…

If the linear frequency shaping bothers you and you'd like to use it logarithmically like other effects, you can always try to make the higher-frequency shapes more audible if you want, by pitching it down after, or use transformative spectral VST to reshape those higher frequencies down, but for sound FX only! Otherwise it throws the harmonics out of line. Still, such "dramatic" and strong effects are made more for sound design anyway, not for the subtle variations I spoke of earlier. It's very good for sound texture manipulations, in this respect. Obviously, such strong effects are for weird sound design only, but I figured I'd say that the output from the FFT Randomizer can be extremely complex and varying as an input source to crazy modulated effects you can use after. So you can use it for anything to post-process it as a complex input texture. If you need a rich and dynamically varying "texture" for your sound design and want to process the hell out of it afterwards for effects, consider the FFT Randomizer at the beginning of the chain too. And you can guarantee that almost nobody will have the same FX, which is unlike if you use an already made texture from a sample pack.

Oh and one more thing I forgot to mention, if you are bothered by the effect "carrying over" to the new chord for instance and you'd rather want it to be a completely different shape (good for transient effects or other fast attack cases), send it a Program Change a few milliseconds before the desired spectral change (and use the MIDI Force Change (ms) parameter to adjust how fast it changes once it receives the Program Change).

But yeah, that's about it! I understand it may seem like a bizarre tool at first glance, but it really got made from my annoyances with lack of proper randomizations and especially phase-control from VSTs. It's one of the most unique effects/VSTs you'll find, sadly enough, even though people might not find it interesting… I'm a perfectionist, so for me it is very important to have this kind of effect, and I personally use it a lot to sculpt up things and make them not sound (or look) the same.

I tried to optimize the code that does the audio processing as much as I could (I didn't bother on other stuff that's not done during audio processing, so the @slider section is not very optimized), by doing speed tests and the like in big loops. I'm actually impressed it uses so little CPU in the end, considering it's not native code. It seems like accessing arrays with [0] is better than indexing, and also arrays are quite slow in JSFX, variables are much faster. Also, interpolating phase changes involves successive sin/cos calculations which are done very optimized, with about 3 multiplications and 2 additions per sin and cos (so 6 and 4 for both), using some mathematical tricks to calculate them from previous two values. Computing sin and cos for every FFT bin would be pretty damn slow otherwise. Note that sin and cos still have to be computed for each of the 513 points, though. (actually, 514 points internally, the last one is a duplicate to aid with the ending smoothness at the highest frequencies)
