Pitchbend Smoother

Pitchbend Smoother

This JSFX script smooths incoming Pitchbend events by slowly going from the previous value to the new value just received, with adjustable randomization and curve shape. Both the shape and the interval can be randomized as well, but only within the range you specify. When using this script, remember that it will introduce lag in the actual pitchbend effects, by its nature. That is in fact very normal for smoothing and is the whole point of it!

For instance, if you have a smooth interval of 500ms, that means it takes half a second to go from current pitchbend to the new pitchbend that was just received. Thus, the "new" pitchbend value will be in effect after half a second, and that's the job of this script: when it sees a pitchbend event, it will smoothly go from the current value to that new one, in the interval specified. If another event happens while it is already in one interval (in other words, for that example, you send another pitchbend event before the 500ms are finished), it will instead go from whatever value it reached towards the new value, in another interval. Without this script, the "new" pitchbend value would be in effect instantly.

The parameters control the density (frequency) of the smoothly generated messages, as well as the interval and curve shape, and their random ranges:

  • Channels: Specifies how many channels to affect, others are passed through (unchanged). It always starts with Channel 1, if you need other configurations, remap and reroute them before sending them to this script.
  • Message Frequency (Hz): This is similar to the parameter with the same name in the Pitchbend Adder: the frequency at which to output the smoothed pitchbends. Note that the value for this is much higher than there: this is because this script smooths the pitchbend events themselves so it needs much more time-resolution in-between pitchbend events or else it would do nothing. In other words, to properly do some smoothing, it needs more density than the pitchbend messages it receives already. Thus the default is 1000Hz, which should be pretty good for most smoothing purposes.
  • Interval Min (ms): How long it takes from the current value to go to the new pitchbend value, in milliseconds (e.g: with no smoothing, this is instant). This defines the minimum interval that is possible, it will never be faster than this.
  • Interval Max (ms): Same as Interval Min but this defines the maximum interval that is possible. The script will randomize an interval between these two parameters on each pitchbend received.
  • Shape Min: This defines the "shape" of the interval. The shape is defined by a normalized value between -1…1. Negative shape values make "fast start" curves that go fast towards the new pitchbend (and slow down at the end), while positive shape values make "slow start" curves (and speed up at the end), with zero shape being completely linear during the interval. Of course, the further away from zero, the more abrupt the curve is in either direction. For example a value of -1 will be very fast at the beginning towards the new pitchbend, slowing down dramatically after that (because it still requires the entire interval to go there). This parameter specifies the "minimum" shape value.
  • Shape Max: Same as Shape Min but this defines the maximum shape value that is possible. Once again, you specify two shape values and the script will generate a random shape between these two values on each pitchbend event. Subtle variations can add some life to the smoothing process, so make use of it wisely.

To help illustrate the shape values, here's a diagram with different colors representing different shape values, and their plotted shapes:

The horizontal axis represents the interval time: left is previous event, right is when it finishes the interval and gets the new event's value. The vertical axis represents the "fade" between the two events; at the bottom we have the value of the previous event, while the top has the next event. As you can see, the shape goes abrupt pretty fast, which is why I recommend using values near zero for most natural effects, but it's up to you. Note also how the simplest shape is a linear fade from the previous event to the next, which is when shape=0.

You can freely automate any parameter except for Channels (which resets everything when changed), including Message Frequency which takes effect almost immediately (well, at least until the next event output). The other parameters don't take effect unless a new interval is started, because the interval length and shape are calculated randomly only when a pitchbend event is received: during the interval it doesn't matter if they change, at least until the next event.

Please note that this script operates entirely in the MIDI domain, so all smoothing is done via calculations on MIDI timings, not per sample. This makes it faster and more precise. Its use should be straightforward, just think of an audio smoother, and how it smooths audio. But instead of audio, you do it on pitchbend. It is also more flexible due to adjustable shape and dynamic variations via randomizations than most other smoothers.

Pitchbend Smoother (Note Reset)

This is the same as the Pitchbend Smoother, but with an extra feature. When there are no notes playing in a given channel and a new Note On is received, with a pitchbend at the exact same time as the note, it will not smooth that pitchbend event (but only that one and it must be exactly when the MIDI note happens). It was made with a very specific purpose in mind. Using Key Tuner, you get pitchbends at the same time as notes to detune them for a specific scale, but what if you want to glide between notes? Without a smoother, you'd end up with instant pitchbend changes for the new note (depending on its detuning) which would bring a small discontinuity in the pitch, since the synth itself will transition between notes at that time (by gliding).

Now, if you were to use the normal Pitchbend Smoother, it would smooth all pitchbend events from the Key Tuner. This is not a problem when you want to do the glide/portamento, but it won't sound right when you trigger a new note without glide (i.e when all notes are off, and the instrument starts a new note). This variant of the Pitchbend Smoother is made for something similar to this case. Its main purpose is to emulate legato in synths (and calculates each channel independently, assuming channel-aware synth or multiple synths out), because when you have no notes playing and a new note is received, the transition is not triggered, thus the pitchbend has to be intact and not smoothed. In this case, and only in this case, it will not smooth it and leave it be, because it assumes there is no transition done by the synth in the first place. Smoothing such a message would cause a sort of bending attack effect.

Obviously, if you have any other note playing on a given channel and a new note is received, the script will work just like the normal version: it will smooth it (that's how legato is triggered in the first place, by holding previous note when triggering a new one). But if you release all notes of a given channel and put a new one then it will be instant: it assumes no legato in that case. This is contrary to the Pitchbend Smoother which will smooth no matter what. Everything else works the same way, though.

—PLEASE NOTE— As mentioned this extra "no smoothing" feature for an event happens only when the pitchbend is at the exact same time (sample) as the note on, and when there are no notes playing except for this new note being triggered. In all other cases, normal smoothing applies. Do remember that this variant of the script is a multi-channel version, so technically it can process several channels at once, and treat each one separately (in terms of notes being active on it or not). This is probably not what you'd expect from a legato synth, which is usually monophonic (i.e one channel only). For that much more common case, see the Pitchbend Smoother Omni (Note Reset) variant below.

Pitchbend Smoother Omni

A variant that's made purely for performance and simplicity. It is the same as the Pitchbend Smoother, but it ignores the input channel and always outputs pitchbends on channel 1. This is basically sort of like a "one channel" version of Pitchbend Smoother, but be sure that the input MIDI has only channel 1, else it might mess things up. If you use only one channel, though, then go ahead and choose it.

Pitchbend Smoother Omni (Note Reset)

This is what you should use for monophonic legato synths in combination with Key Tuner. It is a combination of both variants above, so it ignores the input channel like Pitchbend Smoother Omni, and uses only one channel always, but it does not smooth a pitchbend message if, and only if, it is at the exact same time as a Note On that also happens to "activate" the sound. By "activate", I mean that there is no sound prior to that note — no notes are active, on any channel.

As said, this variant is particularly useful if you use a monophonic instrument with legato/portamento and have a custom scale with Key Tuner or other methods utilizing pitchbend on each note to tune them to that scale. For instance, if the monophonic instrument has half a second of transition between notes, without using this smoother, the pitchbend of the new note will take effect immediately which is not nice as it is a sudden jump in pitch, so a discontinuity. Instead, you set the smoother's interval to around half a second (give or take a few amount for more natural randomness), and its shape probably to something somewhat positive (perhaps 0.2 to 0.4). This shape is well-suited for this task because you end up with very slow pitchbend changes at the beginning of the transition, with most of it being near the end (to help align the new note smoothly). Of course this is just one example, feel free to use it on anything you want.

If you have a synth that handles legato/portamento in a more special way, you'll have to automate the script's parameters accordingly, or get more creative about it. This variant works only with the basic assumption that each new triggered note, when none others are playing, means no transition. Of course, this is the preferred variant to use for most monophonic synths with transitions, because it doesn't keep track of channels, since they're monophonic to begin with. Obviously, legato should be triggered by a new note while the old one is still in effect. If not, it won't smooth it, as it assumes the synth itself won't transition in that case.
